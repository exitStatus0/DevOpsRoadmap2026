# Фактор 1: Хмарне впровадження (Cloud Adoption)

![Cloud Adoption](../../en/01-cloud-adoption/01-cloud-adoption.png)

> **Швидкий старт**
> - **7 днів:** Створіть AWS-акаунт → оберіть актуальний варіант Free Tier / credits під ваш план практики → налаштуйте billing alert → запустіть EC2, S3 та VPC через CLI. Прочитайте «Що НЕ вчити першим».
> - **30 днів:** Виконайте чек-лист початківця → Вправи 1–3 → побудуйте 3-tier архітектуру (Вправа 4).

---

## Старт тут

- **Мінімальний шлях:** Спочатку розберіться з хмарно-агностичними концепціями, сервісами AWS рівня 1 і базовим CLI.
- **Поріг найму:** Вийдіть на рівень **впевнений**, де ви можете зібрати й пояснити невелику production-style архітектуру з IAM, мережею, ALB, RDS, моніторингом та IaC.
- **Залиште на потім:** Control Tower, landing zones, глибокий FinOps, складні multi-account і multi-region схеми.

---

## Чому це важливо у 2026

Переважна більшість компаній уже використовують хмарні сервіси, однак інженерів, які вміють працювати з ними системно й безпечно, усе ще не вистачає.

Хмарна міграція — це не тренд, а **нова норма**. У 2026 році питання вже не «чи переходити в хмару», а «наскільки ефективно ми використовуємо хмару». Компанії, які мігрували 3-5 років тому, тепер оптимізують витрати, впроваджують мульти-хмарні стратегії та автоматизують все.

Що це означає для вас:

- **Попит зростає** — навіть junior-позиції вимагають розуміння хоча б одного cloud-провайдера
- **Зарплати вищі** — cloud-навички можуть суттєво підвищити вашу ринкову ставку
- **Вхідний бар'єр нижчий, ніж здається** — credits і always-free пропозиції зменшують вартість практики, але витрати все одно потрібно контролювати
- **Практична база важливіша за шум** — компанії охочіше спілкуються з тими, хто може показати реальні проєкти й зрозумілі рішення

---

## Яку проблему це вирішує в реальних командах

Уявіть: ваша команда витрачає 3 дні на те, щоб розгорнути новий сервер. У хмарі це займає 3 хвилини. Але тільки якщо хтось знає, як це зробити правильно.

Без cloud-інженера команда стикається з:

| Проблема | Наслідок | Як хмара вирішує |
|----------|----------|------------------|
| Тижні на provisioning серверів | Повільний time-to-market | Сервер за хвилини через API/IaC |
| Невідомі витрати на інфраструктуру | Перевищення бюджету | Pay-as-you-go + інструменти оптимізації |
| Неможливість масштабування | Падіння під навантаженням | Auto-scaling groups, serverless |
| Один дата-центр = один point of failure | Простої | Multi-AZ, multi-region |
| Ручне управління серверами | Людські помилки, нестабільність | Managed services, IaC |

**Реальний приклад:** Стартап із 5 розробників. Без cloud-інженера вони запускали все на одному EC2-інстансі, платили $800/міс і падали раз на тиждень. Після правильної архітектури (ECS + ALB + RDS) — $350/міс, zero downtime, автоматичне масштабування.

## Що потрібно вивчити (ключові навички)

### Хмарно-агностичні концепції (вчіть ПЕРШИМИ)

Ці концепції працюють **у будь-якій хмарі**. Вони — ваш фундамент:

1. **Моделі хмарних сервісів** — IaaS, PaaS, SaaS, FaaS: що коли використовувати
2. **Мережа в хмарі** — VPC, підмережі (public/private), маршрутизація, NAT, peering
3. **Ідентифікація та доступ** — IAM: користувачі, ролі, політики, принцип мінімальних привілеїв
4. **Сховища даних** — об'єктне сховище (S3), блочне (EBS), файлове (EFS), бази даних (RDS)
5. **Обчислювальні ресурси** — VM, контейнери, serverless: коли що обирати
6. **Моніторинг та логування** — CloudWatch / Cloud Monitoring, алерти, дашборди
7. **Управління витратами** — тегування ресурсів, бюджети, Reserved/Spot instances
8. **DNS та балансування** — Route 53 / Cloud DNS, ALB/NLB, health checks

### AWS як перша хмара (рекомендація)

Чому AWS:
- **Широкий enterprise footprint** — багато вакансій і навчальних матеріалів
- **Найширший набір сервісів** — покриває всі сценарії
- **Найкраща документація** та спільнота
- **Free Tier / credits** — корисні для старту, але важливо перевіряти актуальні умови перед практикою

> AWS змінила модель Free Tier у липні 2025 року. Для нових акаунтів тепер важливі обраний план і доступні кредити; старі 12-місячні пільги не варто вважати універсальним сценарієм для курсу 2026.

Ключові сервіси AWS для старту (в порядку пріоритету):

```
Рівень 1 (перші 2 тижні):
├── EC2 — віртуальні машини
├── S3 — об'єктне сховище
├── IAM — управління доступом
├── VPC — мережа
└── CloudWatch — моніторинг

Рівень 2 (тижні 3-6):
├── RDS — управляємі бази даних
├── ECS/EKS — контейнерна оркестрація
├── ALB/NLB — балансувальники навантаження
├── Route 53 — DNS
└── Lambda — serverless

Рівень 3 (тижні 7-10):
├── CloudFormation / Terraform — IaC
├── Systems Manager — управління конфігурацією
├── Secrets Manager — управління секретами
├── SNS/SQS — повідомлення та черги
└── Cost Explorer — оптимізація витрат
```

### CLI та SDK — з першого дня

```bash
# Встановіть AWS CLI та навчіться базовим командам
aws configure
aws s3 ls
aws ec2 describe-instances --output table
aws cloudformation deploy --template-file stack.yaml --stack-name my-stack
```

**Правило:** Якщо ви зробили щось через консоль — повторіть через CLI. Якщо зробили через CLI — автоматизуйте через IaC.

## Що можна пропустити / «найгірший ROI» для початку

Знати, що НЕ вчити — так само важливо, як знати, що вчити. Ось речі, які крадуть час початківців.

### НЕ вчіть першим:

1. **Всі три хмари одночасно** — вивчіть одну глибоко. AWS → потім GCP або Azure для порівняння. Мульти-хмара — це для досвідчених команд, не для початківців.

2. **Сертифікації до практики** — AWS Solutions Architect cert без реального досвіду = красива папірець. Спочатку побудуйте 3-5 проєктів, потім сертифікуйтесь.

3. **Екзотичні сервіси** — AWS має 200+ сервісів. Вам потрібно знати ~15-20. Не витрачайте час на AWS Ground Station або Braket.

4. **Serverless-архітектури першим ділом** — Lambda + API Gateway + DynamoDB — це потужно, але вимагає іншого мислення. Почніть з EC2/ECS, зрозумійте основи, потім serverless.

5. **Детальна оптимізація витрат** — Spot instances, Savings Plans, Reserved Instances — це для етапу, коли ви вже розумієте архітектуру. Не починайте з цього.

### Найгірший ROI:

| Дія | Чому поганий ROI | Що робити замість |
|-----|-------------------|-------------------|
| Читати всю документацію AWS | 200+ сервісів = нескінченно | Вивчати по потребі, проєктно |
| Дивитися 40-годинний курс | Пасивне споживання | Hands-on lab після кожних 30 хв теорії |
| Вчити AWS CDK / Pulumi першим | Потребує знання мови програмування + хмари | Почніть з Terraform або CloudFormation |
| Готуватися до SA Professional | Складний іспит без досвіду = провал | Почніть з Cloud Practitioner або практики |

## Наскільки глибоко занурюватися

### Початківець (2-4 тижні)

- [ ] Створити AWS-акаунт з MFA та billing alert
- [ ] Запустити EC2-інстанс, підключитися по SSH
- [ ] Створити S3-бакет, завантажити файли через CLI
- [ ] Налаштувати VPC з public та private підмережами
- [ ] Створити IAM-користувача з обмеженими правами
- [ ] Зрозуміти різницю між Security Group та NACL
- [ ] Вміти пояснити: що таке Region, AZ, Edge Location

**Тест:** Можете розгорнути веб-застосунок на EC2 з нуля за 30 хвилин? Якщо так — йдіть далі.

### Впевнений (6-10 тижнів)

- [ ] Розгортати інфраструктуру через CLI/IaC (не через консоль)
- [ ] Налаштувати ALB + Auto Scaling Group
- [ ] Використовувати RDS з Multi-AZ
- [ ] Налаштувати CloudWatch алерти та дашборди
- [ ] Зрозуміти мережеву топологію: VPC peering, Transit Gateway
- [ ] Працювати з ECS або EKS для контейнерів
- [ ] Налаштувати CI/CD з CodePipeline або GitHub Actions + AWS
- [ ] Оптимізувати витрати: Right-sizing, Reserved Instances

**Тест:** Можете спроєктувати та реалізувати 3-tier архітектуру (ALB → ECS → RDS) з auto-scaling, моніторингом та IaC? Якщо так — ви впевнений.

### Експерт (3-6 місяців)

- [ ] Мульти-акаунтна стратегія (AWS Organizations, Control Tower)
- [ ] Landing Zone та Security Hub
- [ ] Складні мережеві топології (Transit Gateway, PrivateLink)
- [ ] Оптимізація витрат на рівні організації
- [ ] DR (Disaster Recovery) стратегії та їх імплементація
- [ ] Well-Architected Framework — застосування на практиці
- [ ] Міграція legacy-застосунків у хмару

**Тест:** Можете провести архітектурний ревью проєкту іншої команди та знайти проблеми? Якщо так — ви експерт.

## Як ШІ змінює цей фактор (практичні приклади)

ШІ не замінить cloud-інженера. Але cloud-інженер з ШІ замінить того, хто без.

### 1. Генерація IaC-коду

```
Промпт для ШІ:
"Створи Terraform-конфігурацію для VPC з 2 public і 2 private підмережами
в us-east-1, з NAT Gateway, Internet Gateway та відповідними route tables.
Додай теги для environment=staging."
```

ШІ згенерує 80% коду. Ваша робота — **перевірити, зрозуміти та адаптувати**.

### 2. Аналіз витрат

```
Промпт для ШІ:
"Ось мій AWS Cost Explorer звіт за місяць [вставити дані].
Знайди топ-5 можливостей для оптимізації та оціни потенційну економію."
```

### 3. Дебаг та troubleshooting

```
Промпт для ШІ:
"Мій EC2-інстанс у private subnet не може дістатися до інтернету.
VPC має NAT Gateway. Ось конфігурація route table: [вставити].
Що не так?"
```

### 4. Написання IAM-політик

```
Промпт для ШІ:
"Створи IAM-політику, яка дозволяє читати S3-бакет 'my-app-logs',
писати в CloudWatch Logs групу '/app/production',
та запускати/зупиняти EC2-інстанси з тегом Environment=staging.
Нічого більше — принцип мінімальних привілеїв."
```

### 5. Щоденний workflow

| Задача | Без ШІ | З ШІ |
|--------|--------|------|
| Написати CloudFormation template | 2-3 години | 20-30 хвилин + ревью |
| Розібратися з новим сервісом | Документація 1-2 дні | Пояснення за 10 хвилин + практика |
| Дебаг мережевої проблеми | 1-4 години | 15-30 хвилин |
| Написати IAM-політику | 30-60 хвилин | 5 хвилин + ревью |

**Важливо:** ШІ прискорює, але не замінює розуміння. Якщо ви не розумієте, що ШІ згенерував — ви не зможете дебажити, коли щось зламається.

## Типові помилки та пастки

### Пастка 1: «Кліки по консолі»

**Що це:** Створення всіх ресурсів через AWS Console замість CLI/IaC.

**Чому шкодить:** Ви не зможете відтворити інфраструктуру. Консоль не масштабується. На співбесіді питають про IaC, не про кліки.

**Виправлення:** Правило «консоль тільки для перегляду». Все створення — через CLI або Terraform.

### Пастка 2: «Менталітет сертифікацій»

**Що це:** Витрачати 3-6 місяців на підготовку до AWS SA Associate, не маючи жодного проєкту.

**Чому шкодить:** Сертифікація без практики = знання, які забудуться через місяць. Роботодавці перевіряють навички, не сертифікати.

**Виправлення:** Побудуйте 3 проєкти → потім сертифікуйтеся (буде набагато легше).

### Пастка 3: «Мульти-хмара з першого дня»

**Що це:** Намагатися вчити AWS, GCP та Azure одночасно.

**Чому шкодить:** Жодну хмару не вивчите глибоко. Концепції схожі, але деталі різні — буде каша.

**Виправлення:** Одна хмара глибоко (AWS рекомендовано) → потім порівняння з іншими.

### Пастка 4: «Ігнорування витрат»

**Що це:** Залишати ресурси запущеними після навчання. Не налаштовувати billing alerts.

**Чому шкодить:** Рахунок $500+ за забутий RDS-інстанс — це реально. І це болить.

**Виправлення:**
- Billing alert на $10 — **першим ділом** після створення акаунта
- `aws resourcegroupstaggingapi get-resources` — перевіряти, що все вимкнено
- Terraform `destroy` після кожної навчальної сесії

### Пастка 5: «Тільки managed services»

**Що це:** Використовувати тільки Lambda, DynamoDB, Fargate — не розуміючи, що під капотом.

**Чому шкодить:** Коли managed service не підходить (а це буде), ви не зможете працювати з EC2/ECS/EKS.

**Виправлення:** Почніть з EC2 → ECS → потім EKS/Fargate → Lambda. Знизу вгору.

## Міні-практика (5 вправ)

### Вправа 1: Статичний сайт на S3 + CloudFront (початківець)

**Мета:** Зрозуміти S3, CloudFront, Route 53, SSL.

```
Кроки:
1. Створити S3-бакет для статичного хостингу (через CLI)
2. Завантажити HTML/CSS сайт
3. Налаштувати CloudFront distribution
4. Підключити власний домен через Route 53
5. Налаштувати SSL через ACM
6. Задокументувати архітектуру

Бонус: автоматизувати деплой через GitHub Actions
```

### Вправа 2: VPC з нуля (початківець → впевнений)

**Мета:** Глибоко зрозуміти мережу в хмарі.

```
Кроки:
1. Створити VPC з CIDR 10.0.0.0/16
2. 2 public підмережі (10.0.1.0/24, 10.0.2.0/24) в різних AZ
3. 2 private підмережі (10.0.3.0/24, 10.0.4.0/24) в різних AZ
4. Internet Gateway для public підмереж
5. NAT Gateway для private підмереж
6. Route tables для кожного типу підмережі
7. Bastion host у public підмережі
8. Веб-сервер у private підмережі, доступний через ALB
9. Весь процес — через AWS CLI (жодного кліку в консолі)

Бонус: повторити в Terraform (→ перехід до Фактора 4)
```

### Вправа 3: IAM — правильно з першого разу (початківець)

**Мета:** Зрозуміти IAM на рівні, достатньому для роботи.

```
Кроки:
1. Створити IAM-групи: developers, devops, readonly
2. Створити IAM-політики для кожної групи (принцип мінімальних привілеїв)
3. Створити IAM-ролі для EC2 (доступ до S3) та Lambda (доступ до DynamoDB)
4. Налаштувати MFA для всіх користувачів
5. Створити cross-account роль
6. Протестувати: перевірити, що developers НЕ можуть видаляти S3-бакети

Бонус: автоматизувати через CloudFormation
```

### Вправа 4: 3-tier архітектура (впевнений)

**Мета:** Побудувати production-ready архітектуру.

```
Кроки:
1. ALB у public підмережах
2. ECS Fargate сервіс у private підмережах (мінімум 2 задачі)
3. RDS PostgreSQL Multi-AZ у private підмережах
4. Auto-scaling на основі CPU
5. CloudWatch дашборд з ключовими метриками
6. CloudWatch алерти (CPU > 80%, 5xx > 10/хв)
7. Секрети в AWS Secrets Manager
8. Все через Terraform

Бонус: додати CI/CD з GitHub Actions
```

### Вправа 5: Аналіз та оптимізація витрат (впевнений → експерт)

**Мета:** Навчитися працювати з FinOps-інструментами.

```
Кроки:
1. Увімкнути Cost Explorer та Cost Anomaly Detection
2. Налаштувати тегування ресурсів (Environment, Team, Service)
3. Створити бюджет з алертами
4. Проаналізувати витрати за останній місяць
5. Знайти 3 можливості для оптимізації
6. Розрахувати потенційну економію від Reserved Instances
7. Створити звіт з рекомендаціями

Бонус: автоматизувати звіт через Lambda + SNS
```

## «Сигнали» готовності до роботи (чек-лист)

Ви готові претендувати на junior/mid cloud-позицію, якщо можете:

### Обов'язкове:

- [ ] Пояснити різницю між IaaS, PaaS, SaaS з прикладами
- [ ] Створити VPC з public/private підмережами через CLI
- [ ] Написати IAM-політику за принципом мінімальних привілеїв
- [ ] Розгорнути застосунок на EC2 або ECS
- [ ] Налаштувати ALB з health checks
- [ ] Працювати з S3 через CLI (створення, копіювання, sync, lifecycle)
- [ ] Налаштувати CloudWatch алерти
- [ ] Розгорнути інфраструктуру через Terraform або CloudFormation
- [ ] Пояснити, чому Security Group — stateful, а NACL — stateless
- [ ] Описати, коли використовувати Lambda vs ECS vs EKS
