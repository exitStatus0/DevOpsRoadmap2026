# Фактор 6: Спостережуваність та SRE

![Observability & SRE](../../en/06-observability-and-sre/06-observability-and-sre.png)

> Ви не можете полагодити те, чого не бачите. Кожна команда каже, що має моніторинг — до першого інциденту, коли виявляється, що є дашборди без розуміння. Спостережуваність — це різниця між «щось зламалося» і «сервіс платежів має 12% помилок на ендпоінті /checkout починаючи з останнього деплою». SRE — це дисципліна, яка перетворює цей сигнал на надійні системи.

> **Швидкий старт**
> - **7 днів:** Розгорніть kube-prometheus-stack у вашому K8s-кластері. Напишіть 3 PromQL-запити. Визначте один SLO для сервісу, який ви запускаєте.
> - **30 днів:** Додайте OpenTelemetry SDK до застосунку, відправте трейси до Grafana Tempo та скорелюйте логи + трейси + метрики в одному Grafana-дашборді.

---

## Чому це важливо у 2026

Посада Site Reliability Engineer — одна з найшвидше зростаючих в інфраструктурі. Компанії наймають не просто DevOps-інженерів — вони шукають інженерів, які вміють вимірювати надійність і доводити покращення системи з часом.

У 2026 році:
- **Посади SRE** стабільно пропонують на 15–30% вищу компенсацію порівняно із загальними DevOps-ролями.
- **OpenTelemetry** став другим за активністю проєктом CNCF (після Kubernetes). Кожен вендор тепер підтримує його.
- **SLO-based alerting** замінило порогові сповіщення як стандарт у зрілих організаціях.
- **Distributed tracing** є обов'язковим у будь-якій мікросервісній архітектурі з більш ніж 3 сервісами.

Якщо ви працюєте з Kubernetes (Фактор 2) і не знаєте, як спостерігати за тим, що на ньому запущено — ви наполовину інженер. Спостережуваність — це те, що робить все інше надійним.

> «Моніторинг каже вам, чи працює система. Спостережуваність каже вам, чому вона не працює.»

## Яку проблему це вирішує в реальних командах

**Без належної спостережуваності:**

| Проблема | Реальність |
|----------|-----------|
| Реагування на інциденти | «Щось повільно» — і ніхто не знає де |
| Аналіз першопричини | 2 години пошуку в логах через 5 сервісів |
| Впевненість при деплої | Деплоїте і сподіваєтесь, що нічого не зламається |
| Планування ємності | Здогадки на основі графіків CPU |
| Зобов'язання SLA | «Система була доступна» — але ви не можете це довести |

**З спостережуваністю:**

| Проблема | З спостережуваністю |
|----------|---------------------|
| Реагування на інциденти | Сповіщення: «P99 latency сервісу checkout > 2s протягом 4 хвилин» |
| Аналіз першопричини | Трейс показує: SQL-запит на /checkout займає 1.8s через відсутній індекс |
| Впевненість при деплої | SLO-дашборд показує, що error budget здоровий до і після деплою |
| Планування ємності | Реальне використання ресурсів на сервіс, на namespace |
| Зобов'язання SLA | 99.95% доступності виміряно відносно визначених SLI |

**Реальний приклад:** E-commerce команда мала Prometheus, але без SLO. Під час Чорної п'ятниці сервіс замовлень деградував — але жодне сповіщення не спрацювало, бо всі поди були «запущені». Після впровадження RED-метрик та SLO-based alerting та сама деградація викликала PagerDuty-сповіщення через 8 хвилин після початку інциденту замість 45.

## Що потрібно вивчити (ключові навички)

### 1. Три стовпи: метрики, логи, трейси

Вони не взаємозамінні. Вам потрібні всі три, і ви повинні розуміти, для чого кожен підходить.

```
Три стовпи:
├── Метрики
│   ├── Часові ряди: числа з часом
│   ├── Ідеальні для: дашбордів, сповіщень, трендів
│   ├── Не підходять для: розуміння чому щось сталося
│   ├── Інструменти: Prometheus, Grafana, VictoriaMetrics
│   └── Ключова концепція: кардинальність (вибух міток убиває продуктивність)
│
├── Логи
│   ├── Неструктуровані або структуровані записи подій
│   ├── Ідеальні для: відладки конкретних подій, аудиторських слідів
│   ├── Не підходять для: агрегації, запитів з високою кардинальністю
│   ├── Інструменти: Loki (рекомендовано), ELK Stack, CloudWatch Logs
│   └── Ключова концепція: структуровані логи (JSON > простий текст)
│
└── Трейси
    ├── Розподілений потік запитів між сервісами
    ├── Ідеальні для: знаходження де живе latency в ланцюжку викликів
    ├── Не підходять для: загальних трендів (для цього використовуйте метрики)
    ├── Інструменти: Grafana Tempo, Jaeger, Zipkin
    └── Ключова концепція: поширення контексту трейсу (W3C TraceContext)
```

**Метод RED** (для сервісів):
- **R**ate — запитів на секунду
- **E**rrors — відсоток помилок
- **D**uration — розподіл latency (P50, P95, P99)

**Метод USE** (для ресурсів/інфраструктури):
- **U**tilization — наскільки зайнятий ресурс
- **S**aturation — скільки роботи в черзі/очікуванні
- **E**rrors — події помилок

Починайте з RED для ваших сервісів. Додайте USE для інфраструктури. Ці два методи покривають 80% того, на що потрібно сповіщати.

### 2. OpenTelemetry — Універсальний стандарт

OpenTelemetry (OTel) — стандарт CNCF для інструментування. Він замінює вендорні агенти та SDK.

```
Компоненти OpenTelemetry:
├── SDK (для кожної мови: Go, Python, Java, Node.js, .NET...)
│   ├── Трейси: автоматичне + ручне інструментування
│   ├── Метрики: лічильники, гістограми, gauge
│   └── Логи: кореляція структурованих логів з трейсами
│
├── Collector
│   ├── Отримує телеметрію від ваших застосунків
│   ├── Обробляє: пакетування, фільтрацію, tail sampling
│   ├── Експортує до: Prometheus, Tempo, Loki, Jaeger, Datadog тощо
│   └── Запускається як: sidecar, DaemonSet або окремий сервіс
│
└── Semantic Conventions
    ├── Стандартизовані назви атрибутів (http.method, db.system тощо)
    └── Робить дані сумісними між інструментами та вендорами
```

**Ключова концепція — tail sampling:** Вибіркове зберігання на основі хвоста зберігає 100% трейсів помилок та повільних outliers, і лише частину звичайних трейсів. OTel Collector підтримує правила tail sampling.

### 3. SLI, SLO та Error Budgets

Це концептуальне ядро SRE. Зрозумійте це правильно — і все інше стане на місце.

```
SLI (Service Level Indicator)
  └── Метрика, що вимірює досвід користувача
      Приклади:
      - «Частка HTTP-запитів, що виконуються за < 500ms»
      - «Частка API-викликів, що повертають 2xx або 3xx»

SLO (Service Level Objective)
  └── Ціль для вашого SLI
      Приклади:
      - «99.5% запитів виконуються за < 500ms за 30-денний період»
      - «99.9% API-викликів повертають успіх за 28-денний період»

SLA (Service Level Agreement)
  └── Контракт з вашими користувачами (юридичні/фінансові наслідки)
      Ваш SLO має бути жорсткішим за ваш SLA.

Error Budget
  └── 100% - SLO = бюджет на ненадійність
      «SLO 99.9% = бюджет 0.1% = 43.8 хв/місяць дозволеного даунтайму»
      Коли бюджет здоровий → деплоїте агресивно, впроваджуйте новинки
      Коли бюджет низький   → заморозьте деплої, зосередьтесь на надійності
```

**Чому error budgets важливі:** Вони перетворюють напругу «дев проти ops» на спільну цифру. Error budget — нейтральний арбітр: поки є бюджет, можна рухатись швидко. Коли він вичерпується — уповільнюєтесь. Без сперечань — математика вирішує.

**Інструменти для управління SLO:** Pyrra, Sloth (генерують Prometheus recording rules та alerts з визначень SLO), або вбудований SLO плагін Grafana.

### 4. Сповіщення та чергування

```
Принципи сповіщень:
├── Сповіщайте на симптоми, а не причини
│   ПОГАНО: «CPU > 80%» (причина — може бути нормально)
│   ДОБРЕ: «Error rate > 1% протягом 5 хв» (симптом — користувачі страждають)
│
├── Сповіщайте на burn rate SLO, а не на сирі метрики
│   ПОГАНО: «Кількість помилок > 100»
│   ДОБРЕ: «Burn rate > 2x протягом 1 год» (вичерпає бюджет за 15 днів)
│
├── Кожне сповіщення має бути actionable
│   Якщо не можете діяти прямо зараз — це не дзвінок, це тікет
│
└── Рівні серйозності
    ├── P1: Негайний дзвінок. Вплив на користувачів. Будіть людей.
    ├── P2: Повідомте протягом робочого дня. Деградований досвід.
    └── P3: Створіть тікет. Низький пріоритет. Виправте в наступному спринті.

Інструменти:
├── Alertmanager — маршрутизує Prometheus-сповіщення до одержувачів
│   ├── Маршрутизація: за серйозністю, командою, сервісом
│   ├── Групування: дедублікація пов'язаних сповіщень
│   ├── Silence: вікна обслуговування
│   └── Inhibitions: придушення дочірніх сповіщень при спрацюванні батьківського
│
└── PagerDuty / Opsgenie — управління черговими
    ├── Escalation policies
    ├── Розклади та ротації чергування
    └── Таймлайн інцидентів і інтеграція з postmortem
```

### 5. Управління інцидентами та постмортеми

```
Lifecycle інциденту:
├── Виявлення   — сповіщення спрацьовує (або повідомляє користувач)
├── Тріаж       — оцінка серйозності, перший реагуючий
├── Реагування  — активне усунення (відкат? перезапуск? масштабування?)
├── Вирішення   — сервіс відновлено до SLO
└── Постмортем  — що сталося і як запобігти повторенню

Принципи постмортему:
├── Без звинувачень: системи ламаються, не люди
├── 5 Чому: продовжуйте питати «чому» до знаходження першопричини
├── Action items: конкретні, призначені, з термінами
└── Поширення: публікуйте внутрішньо, щоб всі вчилися
```

### 6. Chaos Engineering (вступний рівень)

Ви не можете довіряти своїй надійності, поки не перевірили її навмисно. Chaos engineering — практика навмисного введення контрольованих відмов.

```
Інструменти:
├── Litmus (CNCF) — K8s-нативний chaos-фреймворк
│   ├── ChaosEngine: визначення мети + експерименту
│   ├── Вбудовані експерименти: pod-delete, network-loss, CPU hog
│   └── Chaos Center: UI для планування та спостереження
│
└── Ручне введення (для навчання)
    ├── kubectl delete pod <name>
    ├── tc qdisc (введення мережевої latency)
    └── stress-ng (навантаження CPU/пам'яті)
```

## Що можна пропустити / «найгірший ROI»

1. **Пропрієтарний APM до OSS** — Datadog і New Relic чудові, але коштують тисячі на місяць. Спочатку вивчіть Prometheus + Grafana.

2. **ELK Stack до Loki** — Elasticsearch потужний, але операційно дорогий. Loki набагато дешевший для K8s-середовищ.

3. **Глибокий chaos до базової спостережуваності** — Без можливості спостереження chaos-експерименти нічого не вчать.

4. **Сповіщення на все** — Більше сповіщень = втома від сповіщень = ігноровані сповіщення. Почніть з 3–5 значущих SLO-based alerts.

## Як ШІ змінює цей фактор

### 1. Генерація PromQL-запитів

```
Промпт: «Напиши PromQL-запит, що розраховує 5-хвилинний error rate
для сервісу 'checkout' по всіх подах, виражений у відсотках від
загальної кількості запитів. Виключи ендпоінти health check (/health, /ready).»
```

### 2. Генерація правил сповіщень

```
Промпт: «Напиши Prometheus alerting rule для SLO burn rate alert.
Сервіс: payments. SLO: 99.9% success rate за 28 днів.
Сповіщення при: burn rate > 14x за 1 год АБО burn rate > 36x за 5 хв.
Включи корисні анотації з посиланням на runbook.»
```

### 3. Чернетка постмортему

```
Промпт: «Напиши blameless постмортем на основі цього таймлайну інциденту:
[вставити таймлайн]
Першопричина: [описати]
Зосередься на системних збоях, а не на помилках людей.
Включи 3–5 конкретних action items.»
```

> ШІ відмінно генерує конфігурацію спостережуваності та запити. Він не може замінити розуміння того, що і чому вимірювати. Спостережуваність — це дизайн-навичка, а не просто конфігурація.

## Типові помилки та пастки

### Пастка 1: Дашборди без сповіщень

**Що відбувається:** Ви будуєте красиві Grafana-дашборди. Інцидент трапляється о 3 ночі. Ніхто не бачить дашборд.

**Виправлення:** Кожна критична метрика повинна мати сповіщення.

### Пастка 2: Забагато рівносильних сповіщень

**Що відбувається:** 50 сповіщень щодня. Інженери перестають на них реагувати. Реальний P1 спрацьовує і ігнорується.

**Виправлення:** Три рівні. P1 будить людей. P2 — Slack. P3 — тікет.

### Пастка 3: Сповіщення на причини, а не симптоми

**Що відбувається:** «CPU > 80%» спрацьовує. Черговий перевіряє — CPU 80%, але сервіс нормально працює. Коли сервіс реально деградує — сповіщення не спрацьовує.

**Виправлення:** Сповіщайте спочатку на симптоми, видимі користувачам: error rate, latency, доступність.

### Пастка 4: Немає trace context у логах

**Що відбувається:** Є трейси, є логи. Але при перегляді трейсу не можна знайти відповідні рядки логів.

**Виправлення:** Включайте trace_id та span_id у кожен рядок логу. OTel SDK робить це автоматично при налаштуванні кореляції логів.

### Пастка 5: Вибух кардинальності метрик

**Що відбувається:** Хтось додає мітку на зразок `user_id` до Prometheus-метрики. Мільйони унікальних часових рядів. Prometheus падає з OOM.

**Виправлення:** Мітки в Prometheus повинні мати обмежену кардинальність. Типові «бомби»: user IDs, IP-адреси, повні URL-шляхи, UUID.

### Пастка 6: SLO без Error Budget Policy

**Що відбувається:** Ви визначаєте SLO, відстежуєте його, але коли error budget вичерпується — нічого не змінюється.

**Виправлення:** Визначте error budget policy до того, як визначите SLO. «Якщо бюджет падає нижче 10% — всі некритичні деплої заморожені.»

## Міні-практика (5 вправ)

### Вправа 1: Prometheus + Grafana + SLO для K8s-сервісу

**Мета:** Повний стек спостережуваності для тестового застосунку.

```
Кроки:
1. Розгорнути kube-prometheus-stack через Helm
2. Розгорнути тестовий застосунок з Prometheus-метриками
3. Написати PromQL-запити: request rate, error rate, P50/P95/P99 latency
4. Створити Grafana-дашборд з RED-панелями
5. Визначити SLO (99.5% success rate) та alerting rule на burn rate
6. Маршрутизувати сповіщення до Slack через Alertmanager

Критерій успіху: дашборд, де одним поглядом видно чи здоровий сервіс
```

### Вправа 2: OpenTelemetry від кінця до кінця

**Мета:** Повний pipeline трейсів від застосунку до Grafana.

```
Кроки:
1. Обрати мову (Go, Python або Node.js)
2. Додати OTel SDK до простого HTTP-сервісу
3. Розгорнути OTel Collector як DaemonSet або sidecar
4. Налаштувати Collector для відправки трейсів до Grafana Tempo
5. Налаштувати Loki для збору логів із trace_id у рядках
6. У Grafana: налаштувати derived fields (клік на trace_id у лозі → перехід до трейсу)

Критерій успіху: можете клікнути на рядок логу і перейти до трейсу цього запиту
```

### Вправа 3: SLO-based alerting з Pyrra

**Мета:** Замінити порогові сповіщення на multi-burn-rate SLO alerts.

```
Кроки:
1. Встановити Pyrra у K8s-кластер
2. Визначити SLO для 2 сервісів (availability + latency)
3. Переконатись, що Pyrra генерує Prometheus recording rules та alerting rules
4. Спровокувати порушення SLO (надіслати помилковий трафік)
5. Спостерігати, як burn-rate alerts спрацьовують на різних вікнах (5m, 1h, 6h)

Критерій успіху: Alertmanager спрацьовує на основі burn rate, а не сирих метрик
```

### Вправа 4: Симуляція інциденту та постмортем

**Мета:** Зламати щось навмисно та задокументувати розслідування.

```
Зламайте щось у тестовому кластері: вбийте залежність, введіть latency,
спровокуйте OOMKilled.
1. Виявіть інцидент через стек спостережуваності (не безпосередньо)
2. Використайте метрики → трейси → логи для знаходження першопричини
3. Напишіть повний постмортем (таймлайн, першопричина, 3+ action items)
4. Хронометруйте: від «сповіщення спрацювало» до «першопричина знайдена»
   має бути менше 15 хвилин

Критерій успіху: blameless постмортем — це матеріал для портфоліо
```

### Вправа 5: Chaos Engineering з Litmus

**Мета:** Контрольоване введення відмов з верифікацією.

```
Кроки:
1. Встановити Litmus у K8s-кластер
2. Визначити ChaosEngine-експеримент: pod-delete для вашого застосунку
3. До експерименту: сформулюйте гіпотезу («сервіс залишиться в межах SLO»)
4. Запустіть експеримент, спостерігайте у Grafana
5. Верифікуйте: чи система поводилась як очікувалось?
6. Задокументуйте: що порушило гіпотезу (якщо щось є) і що б ви виправили

Критерій успіху: chaos-експеримент з порівнянням SLO до/після у Grafana
```

## «Сигнали» готовності до роботи

### Обов'язкове:

- [ ] Розгорнути Prometheus + Grafana + Alertmanager на Kubernetes
- [ ] Писати PromQL-запити для RED-метрик без довідки
- [ ] Визначити SLO та пояснити, що таке error budget нетехнічному стейкхолдеру
- [ ] Налаштувати хоча б одне actionable сповіщення до Slack або PagerDuty
- [ ] Запитувати логи у Grafana (Loki) та фільтрувати за сервісом, подом, рівнем логу

### Бажане:

- [ ] Інструментувати застосунок з OpenTelemetry SDK
- [ ] Корелювати логи + трейси + метрики в одному розслідуванні Grafana
- [ ] Пояснити різницю між методами RED та USE
- [ ] Пояснити tail sampling та чому він кращий за head-based sampling
- [ ] Написати постмортем з першопричиною та action items
- [ ] Знати, коли використовувати SLO burn-rate alerts vs порогові сповіщення

### На співбесіді зможете:

- [ ] Пояснити, що таке error budget та як він змінює динаміку дев/ops

- [ ] Провести розслідування реального інциденту з використанням метрик, логів та трейсів
- [ ] Пояснити різницю між SLI availability, latency та throughput
- [ ] Описати, що означає «blameless postmortem» та чому це важливо
