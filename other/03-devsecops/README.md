# Фактор 3: DevSecOps (Безопасность)

VOICEOVER: В 2026 году безопасность — это не отдельная команда, которая приходит после релиза и всё ломает. Безопасность смещается влево — в пайплайны, в код, в ежедневную работу каждого DevOps-инженера. Если ты не умеешь встраивать безопасность в CI/CD — ты неполный инженер.

ON SCREEN: Схема — Shift Left Security: от «аудит после релиза» к «безопасность в каждом коммите»

---

## Почему это важно в 2026

1. **Рост атак на цепочку поставок (Supply Chain).** SolarWinds, Log4j, codecov — эти инциденты показали, что атаковать проще через зависимости и пайплайны, чем через приложение напрямую. Компании требуют SBOM (Software Bill of Materials) и проверку каждого артефакта.

2. **Регуляторные требования.** GDPR, SOC2, PCI DSS, ISO 27001 — всё больше компаний обязаны соблюдать стандарты. DevOps-инженер, который понимает compliance, стоит значительно дороже.

3. **Зарплатная премия.** DevSecOps-инженеры получают на 20-35% больше, чем «чистые» DevOps. Это одна из самых высокооплачиваемых специализаций в ИТ-инфраструктуре.

4. **Кибератаки каждые 39 секунд.** Это не преувеличение — это статистика. Каждая компания, которую ломают, начинает срочно нанимать людей с навыками безопасности.

---

## Какую проблему это решает в реальных командах

**Проблема: «Секрет в Git-коммите»**

Разработчик случайно закоммитил AWS-ключи в публичный репозиторий. Через 12 минут боты начали создавать EC2-инстансы для майнинга. Счёт: $40 000 за выходные.

**Решение DevSecOps:** Pre-commit hook с `gitleaks` или `trufflehog`. Секреты хранятся в Vault/AWS Secrets Manager. В CI/CD — проверка на утечки секретов.

**Проблема: «Уязвимый Docker-образ в production»**

Образ основан на `ubuntu:20.04` с 147 известными CVE. Никто не проверял.

**Решение DevSecOps:** Trivy в CI-пайплайне. Сборка блокируется, если есть критические CVE. Базовые образы обновляются автоматически.

**Проблема: «Любой под может общаться с любым подом»**

В K8s-кластере нет NetworkPolicy. Скомпрометированный под имеет доступ ко всем сервисам, включая базу данных.

**Решение DevSecOps:** Kubernetes NetworkPolicy по принципу deny-all + разрешение только нужного трафика.

## Что нужно изучить (ключевые навыки)

### Управление секретами

```
Приоритет инструментов:
├── HashiCorp Vault — стандарт индустрии
├── AWS Secrets Manager / Parameter Store — если в AWS
├── External Secrets Operator — секреты в K8s из внешних хранилищ
├── SOPS (Mozilla) — шифрование секретов в Git
└── sealed-secrets — шифрование для K8s/GitOps
```

**Принципы:**
- Секреты НИКОГДА не хранятся в коде или Git
- Ротация секретов должна быть автоматической
- Каждое приложение имеет доступ только к своим секретам
- Аудит доступа к секретам

### Сканирование и безопасность образов

```
Инструменты:
├── Trivy — сканирование образов, IaC, SBOM (рекомендуемый старт)
├── Grype / Syft — альтернатива для SBOM и CVE
├── Cosign (Sigstore) — подписание образов
├── Distroless images — минимальные базовые образы от Google
└── Docker Scout — встроенный в Docker Desktop
```

**Пример Trivy в CI:**
```yaml
# GitHub Actions
- name: Scan Docker image
  uses: aquasecurity/trivy-action@master
  with:
    image-ref: 'my-app:${{ github.sha }}'
    format: 'sarif'
    severity: 'CRITICAL,HIGH'
    exit-code: '1'  # Блокировать сборку при критических уязвимостях
```

### SAST/DAST

```
Статический анализ (SAST) — проверка кода до запуска:
├── Semgrep — универсальный, поддерживает 30+ языков
├── SonarQube — комплексный анализ качества и безопасности
├── Bandit (Python) / Gosec (Go) — специализированные
└── Checkov / tfsec — SAST для Terraform/IaC

Динамический анализ (DAST) — проверка работающего приложения:
├── OWASP ZAP — бесплатный, можно встроить в CI
├── Nuclei — сканер уязвимостей на основе шаблонов
└── Burp Suite (Community) — ручной анализ
```

### RBAC и контроль доступа

- **Kubernetes RBAC:** Role, ClusterRole, RoleBinding, ClusterRoleBinding
- **Принцип наименьших привилегий** — каждый компонент имеет только необходимые права
- **ServiceAccount** — отдельный аккаунт для каждого приложения в K8s
- **IAM Roles for Service Accounts (IRSA)** — в EKS

### Сетевые политики

```yaml
# Запретить весь входящий трафик по умолчанию
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: deny-all-ingress
  namespace: production
spec:
  podSelector: {}
  policyTypes:
  - Ingress

---
# Разрешить трафик только от фронтенда к бэкенду
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-frontend-to-backend
  namespace: production
spec:
  podSelector:
    matchLabels:
      app: backend
  ingress:
  - from:
    - podSelector:
        matchLabels:
          app: frontend
    ports:
    - port: 8080
```

### Безопасность цепочки поставок (Supply Chain Security)

- **SBOM (Software Bill of Materials)** — полный список зависимостей
- **Подписание артефактов** — Cosign/Sigstore для образов
- **Проверка происхождения** — SLSA framework
- **Dependabot / Renovate** — автоматическое обновление зависимостей
- **Admission Controllers** — Kyverno/OPA для проверки политик при деплое

## Что можно пропустить / «худший ROI» для начала

| Ловушка | Почему плохой ROI | Что делать вместо |
|---------|-------------------|-------------------|
| Пентестинг (этичный хакинг) | Отдельная профессия, не задача DevOps | Сфокусируйся на защите: сканирование, политики, секреты |
| Детальные фреймворки комплаенса | SOC2/PCI DSS — работа compliance-команды | Пойми принципы, автоматизируй базовые проверки |
| WAF-тюнинг | Специализированная задача | Знай, что WAF существует, как подключить — достаточно |
| Криптография в деталях | Не нужно для DevOps | Понимай TLS, шифрование at rest/in transit — достаточно |
| Bug Bounty / CTF | Развлечение, не приоритет | Сфокусируйся на defensive security |

## Насколько глубоко погружаться

### Новичок (0-3 месяца)

- Понимание OWASP Top 10 (веб-уязвимости)
- Настройка `gitleaks` как pre-commit hook
- Сканирование Docker-образов с Trivy (локально и в CI)
- Хранение секретов в AWS Secrets Manager или Vault (dev mode)
- Базовый Kubernetes RBAC: создать роль с ограниченными правами
- Настройка HTTPS/TLS для приложения
- **Результат:** Понимаешь базовые угрозы и умеешь автоматизировать простые проверки безопасности

### Уверенный (3-6 месяцев)

- Полный pipeline безопасности в CI/CD: SAST + сканирование образов + проверка секретов
- Kubernetes NetworkPolicy: deny-all + allowlist
- Vault с автоматической ротацией секретов
- PodSecurityStandards (restricted level)
- SBOM-генерация и подписание образов (Cosign)
- Admission Controller (Kyverno): запретить privileged-контейнеры, обязать указывать ресурсы
- **Результат:** Можешь внедрить security pipeline с нуля для проекта

### Эксперт (6-12 месяцев)

- Полный compliance pipeline (SOC2, PCI DSS-совместимый)
- Runtime Security: Falco для обнаружения аномалий
- SLSA framework: полная цепочка доверия от кода до production
- Incident Response: процедуры реагирования на инциденты
- Threat Modeling: моделирование угроз для архитектуры
- Security-as-Code: политики OPA/Kyverno для всего кластера
- **Результат:** Можешь построить security-стратегию для всей организации

## Как ИИ меняет этот фактор (практические примеры)

### 1. Аудит Dockerfile

```
Промпт для ИИ:
"Проведи аудит безопасности этого Dockerfile.
Для каждой проблемы укажи:
- Что не так
- Уровень риска (критический/высокий/средний/низкий)
- Как исправить
[вставь Dockerfile]"
```

### 2. Написание политик Kyverno/OPA

```
Промпт для ИИ:
"Напиши Kyverno ClusterPolicy, которая:
1. Запрещает контейнеры от root (runAsNonRoot: true)
2. Требует указания resource limits
3. Запрещает latest-теги для образов
4. Разрешает образы только из нашего реестра: registry.company.com
Добавь описание и аннотации."
```

### 3. Анализ CVE

```
Промпт для ИИ:
"Trivy нашёл CVE-2024-XXXX в моём образе.
Вот детали: [вставь].
1. Насколько это критично для моего use-case (веб-API, не обрабатывает файлы)?
2. Есть ли workaround, если нельзя обновить зависимость?
3. Как предотвратить такие уязвимости в будущем?"
```

### 4. Генерация NetworkPolicy

```
Промпт для ИИ:
"У меня микросервисная архитектура в K8s:
- frontend (namespace: web) -> backend-api (namespace: api)
- backend-api -> postgres (namespace: data)
- backend-api -> redis (namespace: data)
- Ничего больше не должно общаться друг с другом.
Сгенерируй NetworkPolicy для каждого namespace."
```

**Правило:** ИИ отлично генерирует шаблоны политик, но ВСЕГДА тестируй в staging перед production. Ошибка в NetworkPolicy может заблокировать весь трафик.

## Типичные ошибки и ловушки

### 1. «Безопасность подождёт до аудита»

**Проблема:** Команда откладывает безопасность: «Сначала запустимся, потом добавим». Аудит находит 200+ проблем. Исправление занимает 3 месяца.

**Исправление:** Минимальный security-pipeline с первого дня: gitleaks + Trivy + базовый RBAC. Это 2-3 часа настройки, которые сэкономят месяцы.

### 2. Захардкоженные секреты

**Проблема:** Пароли в `docker-compose.yml`, API-ключи в переменных окружения без шифрования, .env файлы в Git.

**Исправление:**
```bash
# Проверь свои репозитории прямо сейчас:
gitleaks detect --source=. --verbose

# Если нашёл секреты — они УЖЕ скомпрометированы (даже если удалишь коммит)
# Ротируй ВСЕ найденные секреты немедленно
```

### 3. Запуск контейнеров от root

**Проблема:** 80% публичных Docker-образов запускаются от root. Если контейнер скомпрометирован — атакующий получает root-доступ.

**Исправление:**
```dockerfile
# В каждом Dockerfile:
RUN addgroup -g 1001 -S appgroup && adduser -S appuser -u 1001 -G appgroup
USER appuser
```

### 4. «Открытый кластер K8s»

**Проблема:** Нет NetworkPolicy, нет PodSecurityStandards, все поды в одном namespace, RBAC по умолчанию.

**Последствие:** Одна уязвимость в одном поде = доступ ко всему кластеру.

**Исправление:** Deny-all NetworkPolicy + явные разрешения. Отдельные namespace для каждого окружения. RBAC для каждой команды.

### 5. Игнорирование обновлений зависимостей

**Проблема:** Базовый образ не обновлялся 6 месяцев. Накопилось 50+ CVE.

**Исправление:** Настрой Renovate или Dependabot. Автоматические PR на обновление базовых образов. Еженедельная пересборка образов.

## Мини-практика (упражнения для портфолио)

### Упражнение 1: Настройка управления секретами

**Уровень:** Новичок-Средний
**Время:** 3-4 часа

1. Установи HashiCorp Vault (dev mode)
2. Создай секреты для приложения (DB_PASSWORD, API_KEY)
3. Настрой автоматическую ротацию секретов (TTL)
4. Подключи приложение — оно читает секреты из Vault при старте
5. Настрой аудит-лог: кто и когда получал доступ к секретам
6. **Бонус:** Настрой External Secrets Operator в K8s для синхронизации из Vault

**Что покажет работодателю:** Умение работать с секретами правильно.

### Упражнение 2: Security Pipeline в CI/CD

**Уровень:** Средний
**Время:** 3-4 часа

Создай GitHub Actions pipeline с полным security-gate:

```yaml
# Этапы pipeline:
# 1. gitleaks — проверка секретов в коде
# 2. Semgrep — SAST-анализ кода
# 3. Docker build
# 4. Trivy — сканирование образа
# 5. Cosign — подписание образа
# 6. Деплой только если ВСЕ проверки пройдены
```

**Что покажет работодателю:** Понимание security-as-code.

### Упражнение 3: Kubernetes Network Policies

**Уровень:** Средний
**Время:** 2-3 часа

1. Задеплои 3 сервиса в K8s (frontend, backend, database)
2. Примени deny-all NetworkPolicy
3. Разреши только нужный трафик (frontend -> backend -> database)
4. Проверь через `kubectl exec` — попытайся достучаться напрямую из frontend до database. Должно быть заблокировано.
5. Задокументируй схему сетевых политик

**Что покажет работодателю:** Понимание сетевой безопасности в K8s.

### Упражнение 4: Kyverno Policy-as-Code

**Уровень:** Средний-Продвинутый
**Время:** 3-4 часа

1. Установи Kyverno в K8s-кластер
2. Создай политики:
   - Запретить запуск от root
   - Обязать указывать resource limits
   - Запретить `latest` тег
   - Разрешить образы только из доверенного реестра
3. Попробуй задеплоить «плохой» Pod — убедись, что Kyverno блокирует
4. Настрой PolicyReport для отчётов

**Что покажет работодателю:** Умение внедрять автоматические проверки безопасности.

### Упражнение 5: Аудит и Incident Response

**Уровень:** Продвинутый
**Время:** 2-3 часа

1. Установи Falco в K8s
2. Настрой правила: обнаружение exec в pod, чтение sensitive-файлов
3. Подключи алерты (Slack/email)
4. Сымитируй инцидент: `kubectl exec` в pod + `cat /etc/shadow`
5. Напиши постмортем: что произошло, как обнаружили, как предотвратить

## «Сигналы» готовности к работе (чек-лист)

Ты готов закрывать DevSecOps-задачи на работе, если:

- [ ] Никогда не коммитишь секреты в Git (и настроил pre-commit hooks)
- [ ] Можешь настроить Trivy-сканирование в CI/CD pipeline за 30 минут
- [ ] Понимаешь OWASP Top 10 и можешь объяснить каждую уязвимость
- [ ] Умеешь настроить Kubernetes RBAC (Role + RoleBinding для конкретного namespace)
- [ ] Написал хотя бы одну NetworkPolicy (deny-all + allowlist)
- [ ] Работал с HashiCorp Vault или AWS Secrets Manager
- [ ] Знаешь разницу между SAST и DAST и когда применять каждый
- [ ] Твои Dockerfile не запускают приложение от root
- [ ] Можешь объяснить, что такое Supply Chain Security и SBOM
- [ ] Настроил хотя бы одну admission-policy (Kyverno или OPA)
- [ ] У тебя есть проект на GitHub с security pipeline в CI/CD
