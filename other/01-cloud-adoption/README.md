# Фактор 1: Облачное внедрение (Cloud Adoption)

VOICEOVER: Облако — это не «тренд». Это новая реальность. К 2026 году более 75% корпоративных нагрузок работают в облаке. И каждая компания, которая мигрирует, ищет людей, которые умеют это делать правильно.

ON SCREEN: Статистика — рост облачных расходов 2020-2026

---

## Почему это важно в 2026

Три ключевых фактора делают облачные навыки критически важными именно сейчас:

1. **Ускорение миграции.** Компании, которые откладывали переход в облако, теперь вынуждены мигрировать из-за устаревания оборудования, требований регуляторов и конкурентного давления. Это создаёт волну вакансий.

2. **Дефицит кадров.** По данным рынка, на одного квалифицированного облачного инженера приходится 3-5 открытых вакансий. Компании готовы обучать джуниоров — если те понимают базовые концепции.

3. **Мультиоблачность становится нормой.** Крупные компании используют 2-3 облачных провайдера одновременно. Но начинать надо с одного — и глубоко.

---

## Какую проблему это решает в реальных командах

**Проблема:** Команда из 5 разработчиков тратит 2 дня на развёртывание нового окружения. Серверная стойка на исходе, закупка нового железа — 3-6 месяцев.

**Решение с облаком:** Новое окружение разворачивается за 15 минут через Terraform. Масштабирование — автоматическое. Оплата — за использование.

**Проблема:** Стартап растёт с 1000 до 100 000 пользователей за месяц. On-premise инфраструктура не справляется.

**Решение с облаком:** Auto Scaling Group в AWS увеличивает количество серверов автоматически. Бизнес не теряет клиентов.

## Что нужно изучить (ключевые навыки)

### Облачно-агностичные концепции (изучи ПЕРВЫМИ)

Эти концепции одинаковы в AWS, GCP и Azure. Изучив их один раз, ты будешь ориентироваться в любом облаке:

- **Виртуальные сети (VPC/VNet)** — подсети, маршрутизация, NAT, пиринг
- **Вычислительные ресурсы** — виртуальные машины, контейнерные сервисы, serverless
- **Хранилище** — блочное (EBS/Disk), объектное (S3/GCS/Blob), файловое
- **Управление доступом (IAM)** — пользователи, роли, политики, принцип наименьших привилегий
- **DNS и балансировка нагрузки** — Route 53/Cloud DNS, ALB/NLB
- **Мониторинг и логирование** — CloudWatch/Cloud Monitoring, централизованные логи
- **Ценообразование** — модели оплаты, reserved vs. spot/preemptible, калькуляторы стоимости

### AWS как основная платформа для старта

Почему AWS первым:
- Самая большая доля рынка (~32%)
- Больше всего вакансий
- Больше всего документации и сообщества
- Навыки легко переносятся на GCP/Azure

Ключевые сервисы AWS для DevOps (в порядке приоритета):

```
Уровень 1 (обязательно):
├── EC2 — виртуальные машины
├── S3 — объектное хранилище
├── VPC — сети
├── IAM — управление доступом
├── RDS — управляемые базы данных
└── CloudWatch — мониторинг

Уровень 2 (следующий шаг):
├── EKS — управляемый Kubernetes
├── ECR — реестр контейнеров
├── Lambda — serverless
├── ELB/ALB — балансировка нагрузки
├── Route 53 — DNS
└── CloudFormation — IaC (AWS-native)

Уровень 3 (продвинутый):
├── ECS/Fargate — контейнерный оркестратор
├── Systems Manager — управление конфигурацией
├── Secrets Manager — управление секретами
├── CloudTrail — аудит
├── Organizations — мульти-аккаунт
└── Control Tower — управление лендинг-зоной
```

### CLI и SDK — обязательно

```bash
# Установи AWS CLI и научись работать без консоли
aws ec2 describe-instances --filters "Name=tag:Environment,Values=prod"
aws s3 ls s3://my-bucket/
aws iam list-users
```

## Что можно пропустить / «худший ROI» для начала

**НЕ делай это первым:**

| Ловушка | Почему плохой ROI | Что делать вместо |
|---------|-------------------|-------------------|
| Изучать все 3 облака сразу | Поверхностные знания везде = глубокие нигде | Освой AWS на 80%, потом GCP или Azure |
| Начинать с сертификаций | Сертификат без практики = бумажка | Сначала 2-3 практических проекта, потом сертификация |
| Изучать 200+ сервисов AWS | 90% вакансий требуют 15-20 сервисов | Сфокусируйся на core-сервисах выше |
| AWS Console как основной инструмент | Кликать мышкой — не DevOps | CLI + Terraform с первого дня |
| Serverless-архитектура первой | Сложная отладка, специфичная модель | Сначала EC2/ECS, потом Lambda |
| Детальное изучение billing/pricing | Важно, но не приоритет #1 | Базовое понимание + Cost Explorer |

## Насколько глубоко погружаться

### Новичок (0-3 месяца)

- Создать аккаунт AWS (Free Tier)
- Запустить EC2-инстанс, подключиться по SSH
- Создать S3-бакет, загрузить файлы
- Настроить VPC с публичной и приватной подсетями
- Создать IAM-пользователя с ограниченными правами
- Все действия дублировать через CLI
- **Результат:** Можешь объяснить, что такое VPC, подсети, security groups, IAM-роли

### Уверенный (3-6 месяцев)

- Развернуть полное окружение через Terraform (VPC + EC2 + RDS + ALB)
- Настроить Auto Scaling Group с правилами масштабирования
- Реализовать мульти-AZ архитектуру
- Настроить CloudWatch алармы и дашборды
- Работать с EKS (управляемый Kubernetes)
- Реализовать CI/CD с деплоем в облако
- **Результат:** Можешь спроектировать и развернуть production-ready инфраструктуру

### Эксперт (6-12 месяцев)

- Мульти-аккаунтная стратегия с AWS Organizations
- Landing Zone / Control Tower
- Оптимизация затрат (Reserved Instances, Savings Plans, Spot)
- Disaster Recovery и multi-region архитектура
- Compliance и аудит (CloudTrail, Config, GuardDuty)
- Миграция on-premise в облако (реальный кейс)
- **Результат:** Можешь архитектурить решения для enterprise и оптимизировать расходы

### 1. Генерация инфраструктурного кода

```
Промпт для ИИ:
"Сгенерируй Terraform-код для VPC в AWS с 3 публичными
и 3 приватными подсетями в регионе eu-west-1.
Добавь NAT Gateway, Internet Gateway и таблицы маршрутизации.
Используй переменные для CIDR-блоков."
```

ИИ сгенерирует 80% кода. Твоя задача — проверить, понять и адаптировать.

### 2. Анализ затрат

```
Промпт для ИИ:
"Проанализируй этот AWS Cost Explorer отчёт.
Какие сервисы потребляют больше всего?
Предложи 5 конкретных шагов оптимизации."
```

### 3. Диагностика проблем

```
Промпт для ИИ:
"EC2-инстанс в приватной подсети не может достучаться до интернета.
VPC имеет NAT Gateway. Вот конфигурация маршрутных таблиц: [вставь].
Найди проблему."
```

### 4. Подготовка к собеседованию

```
Промпт для ИИ:
"Задай мне 10 вопросов по AWS VPC, которые задают
на собеседовании уровня Middle DevOps.
После каждого моего ответа дай фидбек."
```

**Правило:** ИИ — это ускоритель, но ты должен понимать каждую строку сгенерированного кода. Если не можешь объяснить — не используй.

## Типичные ошибки и ловушки

### 1. «Кликер» — работа только через консоль

**Проблема:** Ты умеешь создать EC2 через AWS Console, но не можешь воспроизвести это через CLI или Terraform.

**Последствие:** На собеседовании просят написать Terraform — и ты в ступоре. В реальной работе всё делается через код.

**Исправление:** Правило 80/20 — 80% действий через CLI/Terraform, 20% через консоль (только для мониторинга).

### 2. «Коллекционер сертификатов»

**Проблема:** 3 сертификата AWS, 0 реальных проектов на GitHub.

**Последствие:** Собеседования проходишь теоретические, но заваливаешь практические.

**Исправление:** Сначала портфолио из 3-5 проектов, потом сертификация как подтверждение.

### 3. «Мультиоблачный теоретик»

**Проблема:** Знаешь названия 50 сервисов в 3 облаках, но не можешь настроить VPC ни в одном.

**Исправление:** Выбери одно облако. Освой его на уровне «могу развернуть production». Потом — второе.

### 4. «Забыл про деньги»

**Проблема:** Оставил запущенные ресурсы в AWS. Счёт на $500 в конце месяца.

**Исправление:**
- Настрой Billing Alerts с первого дня
- Используй `aws-nuke` или скрипт для очистки ресурсов
- Всегда проверяй Free Tier лимиты
- Используй `terraform destroy` после экспериментов

### 5. «Игнорирование IAM»

**Проблема:** Всё делаешь под root-аккаунтом или с `AdministratorAccess`.

**Последствие:** На реальной работе тебе дадут ограниченные права, и ты не будешь понимать, почему ничего не работает.

**Исправление:** С первого дня создай отдельного IAM-пользователя. Используй принцип наименьших привилегий.

## Мини-практика (упражнения для портфолио)

### Упражнение 1: Статический сайт на S3 + CloudFront

**Уровень:** Новичок
**Время:** 2-3 часа

1. Создай S3-бакет для хостинга статического сайта
2. Загрузи простой HTML/CSS сайт (портфолио или лендинг)
3. Настрой CloudFront-дистрибуцию перед S3
4. Подключи собственный домен через Route 53
5. Включи HTTPS через ACM (бесплатный сертификат)
6. **Бонус:** Автоматизируй деплой через GitHub Actions

**Что покажет работодателю:** Понимание S3, CDN, DNS, SSL/TLS.

### Упражнение 2: VPC с нуля через CLI

**Уровень:** Новичок-Средний
**Время:** 3-4 часа

```bash
# Создай VPC
aws ec2 create-vpc --cidr-block 10.0.0.0/16

# Создай публичную подсеть
aws ec2 create-subnet --vpc-id <vpc-id> --cidr-block 10.0.1.0/24

# Создай приватную подсеть
aws ec2 create-subnet --vpc-id <vpc-id> --cidr-block 10.0.2.0/24

# Настрой Internet Gateway, NAT Gateway, таблицы маршрутизации
# Запусти EC2 в каждой подсети
# Проверь доступность
```

Всё через CLI. Потом задокументируй архитектуру в README.

**Что покажет работодателю:** Глубокое понимание сетей в облаке.

### Упражнение 3: IAM — правильная настройка

**Уровень:** Средний
**Время:** 2-3 часа

1. Создай 3 IAM-группы: `developers`, `devops`, `readonly`
2. Напиши кастомные IAM-политики для каждой группы
3. Настрой MFA для всех пользователей
4. Создай IAM-роль для EC2, которая имеет доступ к определённому S3-бакету
5. Протестируй, что ограничения реально работают
6. **Бонус:** Напиши всё в Terraform

**Что покажет работодателю:** Понимание безопасности и принципа наименьших привилегий.

### Упражнение 4: Отказоустойчивая архитектура

**Уровень:** Средний-Продвинутый
**Время:** 4-6 часов

1. Создай ALB (Application Load Balancer)
2. За ним — Auto Scaling Group с min=2, max=6
3. EC2-инстансы с простым веб-приложением
4. RDS в Multi-AZ режиме
5. CloudWatch алармы при CPU > 70%
6. Всё через Terraform
7. Протестируй: убей инстанс — убедись, что ASG поднимает новый

**Что покажет работодателю:** Понимание HA, масштабирования, мониторинга.

### Упражнение 5: Оценка и оптимизация затрат

**Уровень:** Продвинутый
**Время:** 2-3 часа

1. Разверни инфраструктуру из упражнения 4
2. Используй AWS Cost Explorer для анализа
3. Замени On-Demand инстансы на Spot (где возможно)
4. Настрой S3 Lifecycle Policies
5. Напиши отчёт: «Как я сэкономил X% на этой архитектуре»

## «Сигналы» готовности к работе (чек-лист)

Ты готов откликаться на вакансии Junior/Middle Cloud/DevOps, если:

- [ ] Можешь создать VPC с публичными и приватными подсетями **по памяти** (CLI или Terraform)
- [ ] Понимаешь разницу между Security Group и NACL
- [ ] Можешь объяснить, как трафик идёт от пользователя до EC2 в приватной подсети (через ALB)
- [ ] Настроил IAM с кастомными политиками (не только `AdministratorAccess`)
- [ ] Развернул хотя бы одну отказоустойчивую архитектуру (Multi-AZ)
- [ ] Работаешь преимущественно через CLI и Terraform, а не через консоль
- [ ] Можешь объяснить ценообразование EC2 (On-Demand, Reserved, Spot)
- [ ] У тебя есть 2-3 облачных проекта на GitHub с README и Terraform-кодом
- [ ] Можешь ответить на вопрос: «Расскажи про последний проект, который ты развернул в облаке»
